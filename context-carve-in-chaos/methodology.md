# 上下文雕刻方法论

## 这是一份什么

一份**识别和构建信息结构**的方法。不是手册——手册绑定特定场景，这里要的是换了场景、换了名字之后，结构本身还能被认出来。

适用场景：写 skill、写文档、审文档、设计系统间接口。

## 种子结构

任意由符号关系组成的系统（文档、代码、对话、架构），其稳定性和清晰度取决于三个维度：

**密度：** 一段信息里，有多少字节在做有用功。用更少的话说更多的事 → 密度高。废话、重复、符号装饰 → 密度低。

**闭环：** 信息单元之间是否两两可互查。A 提到 B，B 也得能指回 A。单向引用是开环，悬空。

**一致性：** 同一个东西在不同地方用不同名字说，是否从名字能认出是同一个东西。认不出来 → 不一致。

## 怎么看

不要套术语。先读。哪里别扭就停下来。别扭的地方就是结构断裂的体感。

三个信号：

| 体感 | 对应的断裂 |
|------|-----------|
| "这个词到底指什么？" | 开环——能指找不到所指 |
| "这不是跟前面说的一样吗？" | 冗余——同一个所指挂在多个能指下，没说它们是等价的 |
| "前后说的是一回事吗？" | 矛盾——同一个所指指向了互相冲突的描述 |

## 怎么修

顺序很重要。先建背景，再锚概念，最后收细节。顺序反了会在修的过程中制造新的断裂。

1. 先让读的人有立足点——入口文档，基础定义
2. 再让每个词能落地——要么给定义，要么给例子，要么从上下文能推导
3. 再清理——合并等价说法，删掉解释自己的解释
4. 再压缩——删完重读，还能再删吗？

## 怎么验

修完之后，用修之前同样的三个问题再量一遍。自己用的尺子必须先量自己。第一次修的时候一定漏东西——把漏的记下来，下次的"先建背景"就更厚了。

## 怎么移植

上面这些描述绑定了一个领域（文档）。但这个结构本身不绑定领域。换个领域——比如代码架构、对话设计——怎么让结构浮现？

不是换名字套术语。是换例子。同一组问题（密度？闭环？一致性？）在不同领域，用不同的材料回答。问题不变，答案变。结构被问题锚定，不被答案锚定。
