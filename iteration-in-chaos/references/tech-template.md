# Tech 文件格式

文件名：`techs/{name}.md`

```markdown
# 技术名
## status: [draft | active | rejected]
## 覆盖维度
## 边界
- 覆盖: [审什么]
- 不覆盖: [不审什么]
- 失效条件: [什么情况下不管用]
## 测试案例
| 输入 | 应用 | 预期输出 | 实测结果 |
## 效果对比
| before | after | 是否改善 | 证据 |
## 验收标准
## 盲区
## 来源
## 生效记录
| 日期 | 项目 | 通过? | 失败根因 | 备注 |
```

## 验证门三要素

所有技术初始 `status: draft`。变为 `active` 需要：

| 要素 | 要求 |
|------|------|
| 边界 | 覆盖什么、不覆盖什么、失效条件 |
| 测试案例 | 至少 1 个：输入 → 应用 → 预期输出 → 实测结果 |
| 效果对比 | before/after：不用 vs 用，差异在哪 |

```
draft → 匹配场景时实测 → 记录 before/after
  ├── 正向 → active，进管线
  ├── 不明确 → 保持 draft
  └── 负向（反优化） → rejected → archive/
```

用户提出的方向也走验证门。直觉是触发信号，不是豁免。
