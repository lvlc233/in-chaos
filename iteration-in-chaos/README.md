# iteration-in-chaos

在人类和 agent 共同执行项目（代码、文档等）时，将人类反复提出或认可的迭代方案，从散落的对话中提取为结构化的文件。agent 之后遇到符合该方案的内容，可以自动执行迭代，无需人类二次指导。

## 概念

### 用户反馈 {不错}

人类在项目中给 agent 的纠正、原则声明、方向调整。一句纠正算一次反馈，多次同类纠正累积成模式。反馈的信息来源是项目内的交互过程。

关键判据：反馈里能不能提取出一条 agent 之后可以自动执行的规则或流程。

### 迭代对象 {可以}

当前正在迭代的目标——一个 skill、一组关联文件、或项目中的某个模块。同一轮迭代内，迭代对象保持不变。

### 迭代方案 (tech)

从用户反馈中提取的、agent 可独立执行的方案{清晰}，落盘为 `techs/{name}.md`。每个方案包含三要素：

- **边界**：这个方案覆盖什么场景、不覆盖什么场景（例如：覆盖 skill 文件结构检查，不覆盖内容质量评测）{标准}
- **验证方式**：方案执行后怎么判断通过还是失败（例如：检查项全部通过算 pass，任意一项失败算 fail）
- **效果记录**：方案实际应用前后的对比（例如：用了之后文件结构从 X 变为 Y，Y 是否符合预期）

三类方案：检查方法（怎么发现同类问题）、执行流程（按什么步骤修）、设计约束（什么情况下不改）。{嗯，也是定义}

### 状态

本 skill 内部的 8 个执行阶段，整体流程是：反馈积够了 → 整理成方案文件 → 方案进执行列表 → 逐个跑 → 跑完验证 → 验证通过则方案保留，下次自动用；失败则排查是方案本身不对还是用错地方 → 修正或废弃。

从用户往下读的角度，前 4 个是直接可触发的，后 4 个是顺着前面跑进去的：

**可直接触发的：** new-task（开始新迭代）、feedback-received（收到用户纠正）、accumulated（反馈积累够了）、revalidate-preference（检查已有方案还有效吗）

**顺着前面跑进去的：** crystallizing（把积累的反馈做成方案文件）、applying-tech（执行方案中的步骤）、tech-failed（方案执行失败了）、tech-rejected（方案根本不好用，废弃）

## 文件说明

```text
iteration-in-chaos/
  README.md         ← 本文件，人类了解背景和结构
  SKILL.md          ← agent 加载的执行指令（状态机、路由、规则）
  techs/            ← 迭代方案文件，agent 读取后自动执行
    index.md
    *.md
  preferences/      ← 不同场景下先用哪个方案、什么顺序
    index.md
    *.md
  log/              ← 用户反馈记录，每一条含提取的原则
    _proposals/     ← 积累够了但还没确认要不要做成方案的候选
    *.md
  archive/          ← 已废弃的方案
  references/       ← agent 创建新方案/日志/偏好时的格式参考（模板）
    log-template.md
    tech-template.md
    preference-template.md
  events/           ← 运行记录（自检偏离等）
  TODO.md           ← 待修问题追踪
```
