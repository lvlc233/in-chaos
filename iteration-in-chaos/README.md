## 核心概念

### 用户反馈
用户在{项目的}迭代过程中给{agent或项目本身的}的纠正、原则声明、方向调整。一句话也算，累积成模式也算。关键判据：**反馈里能提取出可复用的原则**。

### 迭代对象
当前正在迭代的目标——可以是一个 skill、一个系统组件、或一组关联文件。同一个迭代会话内，迭代对象固定。

### 技术 (tech){这个翻译合理，因为在文件夹中会使用tech而非技术，但是，技术这个词仍然不好，只是说这里的补充是正确的}
从用户的反馈中提取的**可复用方案**，落盘为 `techs/{name}.md`。一个技术{同上}包含三个要素：边界（覆盖/不覆盖）{覆盖什么呢？}、测试案例（怎么验证）{那测试案例长什么样子呢？}、效果对比（用了之后改了什么）。三类技术：检查方法、执行流程、设计约束。{未定义}{但是这个xxx（）的补充结构是正确的}

### 偏好 (preference) {这个，其实，结构上可以，但是我作为创作者知道，这一部分是不要的}
用户在**多个可用技术中做编排选择**——什么维度用什么技术、什么顺序。存为 `preferences/{dimension}.md`，必须指向日志证据。

### 管线 (pipeline){这一部分是不要的}
偏好里的技术执行序列，带 `current_step` 标记支持断点续跑。一条管线是一组有序的技术。
