# blind-spot-driven-design
## status: active
## 覆盖维度
skill-design, review-methodology
## 边界
- 覆盖: 用外部审核暴露的技术盲区作为新技术的需求规格，驱动技术体系自演进
- 不覆盖: 单次缺陷修复（那不是盲区，是已知 bug）
- 失效条件: 外部审核只提了执行错误（如拼写、格式），没暴露技术盲区
## 测试案例
| 输入 | 应用 | 预期输出 | 实测结果 |
|------|------|----------|----------|
| superpowers code review 发现 state-machine-check 盲区（不检查 transitions 闭环） | 用盲区作为配套技术的需求规格 | 识别出需要新 tech 或扩展 state-machine-check 的盲区声明 | 通过——产出了本 tech |

## 效果对比
| before | after | 是否改善 | 证据 |
|--------|-------|----------|------|
| 技术盲区被当作免责声明（"这部分管不了"），无人追踪 | 每个盲区都是对配套技术的需求规格，盲区 = 待填补的接口 | 改善 | 本次双 agent 验收中 evaluating 发现的自举双标 → 驱动 5 项修复，盲区从免责声明变为可执行需求 |
## 验收标准
- 每次外部审核后，列出当前技术体系中哪些盲区被暴露
- 每个暴露的盲区有明确的处理：创建新技术 / 扩展现有技术 / 补充盲区说明
- 新技术过验证门后才进管线
## 使用说明
1. 外部审核报告 → 逐条缺陷映射到当前 techs/ 中各项技术的盲区声明
2. 区分：已知盲区内（技术声明了不审这块，审核恰好命中）→ 该盲区是正确的，审核完成了配套；已知盲区外（审核发现了技术未声明的盲区）→ 需要扩展该技术的盲区声明；全新领域（审核发现了当前技术体系完全不覆盖的维度）→ 需要新 tech
3. 创建新 tech 或扩展时，必须标注"来源 = 某次外部审核暴露的盲区"
4. 新 tech draft 后，在下次任务中实测，走验证门
## 盲区
- 不判断外部审核本身的质量——坏的审核也可能"暴露盲区"，但那是噪音而非需求
- 不替代内容审查——盲区驱动的是技术体系的结构性补全，不覆盖执行细节
- 对纯内部迭代（无外部视角输入）不适用
