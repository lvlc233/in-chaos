## status: active

skill-design
## 边界
- 覆盖: 检查 skill/指令是否以状态机形式组织（状态→动作的映射表），而非线性流程（step1→step2）
- 不覆盖: 状态机内部动作的质量（由内容审查覆盖）
- 失效条件: 当 skill 本身是纯线性任务（如一次性配置脚本）时，状态机结构不可用

## 验收标准
- 目标 skill 中能找到明确的状态→动作映射（不需要 agent 推断步骤顺序）
- 任何合法场景都能作为独立入口点进入（不需要前提步骤）
## 使用说明
状态表不止是组织结构——把内容放进状态表的过程本身就是编辑过滤器：写不进状态表的内容自动暴露为参考说明（应进 `references/`），不是执行指令。用这个技术时，先把所有内容往状态表的"触发条件/执行动作/后续状态"三列里填，填不进去的考虑删除或移到 references/。
## 盲区
- 不检查状态定义是否完备（是否有遗漏的状态）
- 不检查各状态动作集是否正确
- 不检查 transitions 是否形成闭环
- 对非 skill 类文档（如 README）不适用
