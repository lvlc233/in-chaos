# 8 维审视 rubric

> 每维四栏:**① 检查问题 ② 好的样子(标尺) ③ 坏味道(red flags) ④ 严重度指引**。
> 标尺锚点:superpowers / writing-skills(好 skill 评分卡)、video-use(长流程落盘范式)。
> 这份 rubric 是**演进中**资产——每次真实审视后把新坑回填进来(见 `../CHANGELOG.md`)。

> **审前先给 skill 归类**(决定哪些维度该重、哪些会降级):
> - **长链路有状态型**(分析→搜索→编排→出片这种多阶段携带中间态)→ 维度 3 最致命。
> - **短链路工具型**(2–4 步闭环、近乎无状态、靠脚本/命令干活)→ 维度 8 最致命,维度 3 多半降级。
> - **纯指导型**(无脚本,只给方法论)→ 维度 8 可能 N/A。
> - **重构后复审型**(被审对象有前一版 + 已知缺陷清单)→ **先做修复回归**:逐条旧缺陷判"真修好 / 部分 / 倒退",再**遗留扫描**(旧契约里有没有跟新支柱冲突、没翻面的措辞——重构最易在这翻车;**换名词式重构的可机检四查见 ▸专项:重构 / 换名词式改造的遗留扫描**),然后才逐维。

## 维度 1 — 体量与注意力预算

**① 检查问题**
- SKILL.md 多少行?加载进每次对话要吞多少 token?
- 哪些内容"每次都必须在场",哪些"按需才读"?二者分清了吗?
- references 有没有把重内容下沉?skill 目录里有没有混进无关大文件?

**② 好的样子**
- superpowers SKILL.md 普遍 100–700 行;writing-skills 主张 frequently-loaded <200 词、其他 <500 词。
- 细节下沉到 references / 工具 `--help` / cross-reference 到别的 skill。
- video-use:348 行骨架 + manim references 按需读,主文件可一次读完。

**③ 坏味道**
- SKILL.md 动辄上千行,把所有细节都塞进主文件。
- references 该拆不拆,或拆了又在主文件重复(冗余)。
- 把无关大文件(横幅图片、整套无关子 skill、LICENSE、install 文档)塞进 skill 目录拉高体量与认知混乱。
- 体量没拆"自写 vs vendor 进来"分开数 → 易错判病灶。

**④ 严重度**:主文件臃肿 🟠;无关大文件拖累 🟡。

## 维度 2 — 单一职责 vs 大杂烩

**① 检查问题**
- 这个 skill 到底负责几件事?多场景是否共享同一契约 / 流程,还是被硬塞进一张图?
- **有没有明确的「不做什么」边界声明?**(固定当成一个正面打勾位记录——这一项一眼能判)
- 职责名实相符吗(名字说 A,正文在干 B)?
- **分层体系的层间重叠**(若被审对象是分层 / 多模块):各层 / 模块的**判定 territory 互斥吗**?同一个观测量会不会被两层重复处理 / 打分、且没声明分工?

**② 好的样子**
- writing-skills 专注"写 / 测 skill";video-use 专注"对话式剪视频"。
- 显式列"不归我们管 / 明确不做"边界(git-sync `SKILL.md:26-30` 干脆列了 ❌ 不做 conflict/分支/push/stash,是范本)。

**③ 坏味道**
- 一个 skill 通吃差异巨大的多场景,却塞进同一张流程图,分支用 `├───` 并存增加认知负载。
- 没有"不做什么"边界声明,读者不知道它到哪儿为止。
- 多场景里某个分支完全依赖外部把状态拼好(与维度 3 联动)。

**④ 严重度**:🟡(除非多职责耦合到互相破坏 → 🟠)。

## 维度 3 — 状态如何跨上下文压缩 / 跨步骤传递 ⭐(长链路有状态型最致命)

**① 检查问题**
- **先确认归类**:这是长链路有状态型还是短链路工具型?**短链路工具型本维大概率降级,别硬套 🔴**——它根本没有需要活过压缩的状态。
- (长链路型)关键中间态(分析结论、ID、时间戳、计划、verified 结果)靠什么活过**上下文压缩 / 跨 session / 异步派发**?
- 是**落盘成可重读产物**(健壮)还是只在对话 in-memory(脆弱)?启动 / 续跑是"先读落盘"还是"信任外部拼好历史"?
- 试金石:**"如果在第 K 步上下文被压缩,前面算出来的东西还在吗?换个 turn 还读得到同一个文件吗?"**

**② 好的样子**
- video-use:`project.md`(每会话追加决策日志)+ `edl.json`(剪辑状态)+ `takes_packed.md`(转写主视图),
  **启动先读 project.md** 并总结上次会话;转写缓存幂等。

**③ 坏味道**
- "in-memory,不落盘 / 不存中间文件";"信任 agent runtime 已拼好历史";"不主动恢复 session"。
- 把"上下文窗口截断 / 摘要 / 拒绝"划成"不归我们管"。
- 长链路(N 步)却**无任何阶段产物落地**;甚至显式禁用可复用的落盘层("中间产物不读不动不依赖")。
- **落了盘 ≠ 读得回**:落盘路径用相对路径 / 依赖 cwd 稳定 → agent runtime 的 cwd 在 turn 间漂移时,各 turn 写到不同绝对目录,**看似落盘实则回退/续做时读空**(比没落盘更隐蔽)。落盘路径应首跑解析成绝对路径并自存。

**④ 严重度**:**长链路 + 不落盘 = 🔴**(对压缩零容错,生产必现"前半段结论丢失")。短链路 / 无状态 → 降级 🟡 或 N/A。

## 维度 4 — 控制流是否显式

**① 检查问题**
- 有没有决策树 / flowchart / 状态机?**决策骨架与执行细节分离了吗?**
- 分支 / 失败 / 回退 / "这步不能和那步颠倒"的顺序约束,是显式编码还是埋在散文里靠模型自己悟?

**② 好的样子**
- superpowers graphviz digraph:`diamond`=决策、`box`=行动、`doublecircle`=终态,**只画骨架**,边标 `yes/no`;细节留正文 / references。
- video-use:8 步流程是可逐条扫的**清单**。

**③ 坏味道(双向,两端都翻车)**
- **塞太多**:一个大 ASCII / 文字块同时承载控制流 **+** 每步的输入 / 动作 / 输出 / 门 / example / 兜底
  (writing-skills 反模式 **"❌ Code in Flowcharts"**)——流程图被当成正文容器。
- **该有却没有**:有多分支 / 顺序约束却**完全无骨架图 / 状态机,全靠散文**让模型自悟主干——
  压缩时顺序约束("这个顺序不能颠倒")最易整段丢失或错位。"没塞任何东西进图"不等于没问题。

**④ 严重度**:🟠(控制流不可机读、压缩时整块易丢、人改一步易错位)。

## 维度 5 — 强制力分层 + 防幻觉集中

**① 检查问题**
- hard rule(违反即崩)和 soft guidance(可灵活)在排版上区分得开吗?有没有 Iron Law / CORRECTNESS 区?
- 有没有显式标注 **Rigid vs Flexible**?有没有两条"硬规则"语义上互相打架却都没标级别?
- 防幻觉规则是**集中**成"红旗区 / 反模式表",还是散落在各 step 里?

**② 好的样子**
- superpowers:Iron Law 用代码块宣示("NO X WITHOUT Y");集中 **Red Flags 清单** + **借口-现实对照表**;
  `<EXTREMELY-IMPORTANT>` / `<HARD-GATE>` 标签;Rigid/Flexible 显式标注。
- video-use:12 条 Hard Rules(每条带原因+后果)vs artistic freedom **分离**。
- (video-remix 的 CORRECTNESS vs CRAFT 两层分离 = 此项正面做法。)

**③ 坏味道**
- hard 和 soft 混排,无视觉区分,读者分不清哪条碰了即崩;两条硬规则互相软化却都没标级别。
- 防幻觉叮嘱("别编 / 亲眼看 / 用库里的词")散在 N 个 step 里——**上下文压缩时最先被摘**。
- 没有一处集中的"这些念头说明你在偷懒 / 自欺"清单。

**④ 严重度**:hard/soft 不分 🟠;防幻觉散落 🟡(但会**放大维度 3 的 🔴**:压缩先丢防线)。

## 维度 6 — 可组合性与下游感知

**① 检查问题**
- 复用别的 skill 是 **cross-reference** 还是**整目录拷贝(vendor)**?拷贝带没带无关物?
- **同仓 / 同域有没有职责重叠的现成 skill 没复用?**(未复用比 vendor 拷贝更隐蔽——得主动扫同目录其他 skill 才发现;典型:自己重写了一套别的 skill 已有的逻辑)
- skill 构造的输出喂给下游(另一个 skill / 服务 / agent)时,有没有在**同一步**告知下游的处理机制 / 契约?
- **落盘归属消歧**:若有两套落盘(自己的 vs 复用的下游 skill 私有的),两边边界讲清没?消歧措辞**自身**一致吗?(常见 bug:消歧时把某产物归错边,导致同一文件内"落盘/不落盘"自相矛盾)
- **测试 vs 被测漂移**(若被审对象是测试 / eval):expected / 契约是 cross-ref 被测 skill 的单一源头,还是拷了一份会各自漂移?**被测 skill 重构后,测试翻面了吗**?(典型:skill 加了字段,evals 的 expected 没跟 → 测了过时行为)
- **提炼版 ↔ 母体漂移**(若被审对象是从某母体提炼的通用版):母体(源头)进化后,提炼版翻面了吗?提炼版应**回链母体**、标"母体变更需同步本处"。(与"测试 vs 被测"同构,方向相反——被提炼方是源头)
- **配对双向核验**:若 skill 声明与另一 skill **配对 / 互补**,对方**回链了吗**?A→B 引了、B→A 没引 = 配对契约只落一半。
- 引用是松耦合(`[[link]]` / "按需读 X")还是硬编码路径?

**② 好的样子**
- writing-skills 主张 cross-reference("Reference other skill")。
- 下游感知:把 vendor 模式 / wrapper / 端点白名单 / 谁负责翻译(mapper)讲清楚,输出字段标数据来源。

**③ 坏味道**
- vendor 整目录且含无关大文件 → 体量拖累 + **双版本漂移**(原版升级副本不跟)。
- 同仓已有职责重叠 skill 却另起炉灶重写(看不见的重复 + 漂移)。
- 构造输入喂下游却**不告知下游怎么处理**(下游感知缺失 → 下游静默误用)。
- 到处硬编码路径而非松引用。

**④ 严重度**:漂移 / 维护 / 重复 🟡;下游感知缺失可致 🟠(静默误用难排查)。

## 维度 7 — Description = 触发条件(CSO)

**① 检查问题**
- description 只写"**何时用**(触发条件 / 症状 / 关键词)"吗,还是混进了"做什么 / 能力清单 / 流程"?
- 第三人称?关键词是否覆盖错误信息 / 症状 / 工具名?技术专属是否显式标出?

**② 好的样子**
- `Use when implementing any feature or bugfix, before writing implementation code`(只写触发,不写流程)。
- 关键词覆盖症状("flaky"/"race condition")而非具体语言细节("setTimeout")。

**③ 坏味道**
- description **概括了 workflow / 能力清单**("支持 X、Y、Z")→ writing-skills 实测的陷阱:Claude 照着 description 走、**跳过 skill 正文**。
- 第一人称;太抽象("for async testing");提了技术但 skill 并不专属该技术。
- 正文又把 description 的触发词整段重抄一遍(冗余)。

**④ 严重度**:⚪–🟡(影响触发准确性与"是否真去读正文")。

## 维度 8 — 执行层正确性 / 失败模式 ⭐(工具型 / 脚本型最致命)

> 前 7 维都在审**信息怎么组织**,默认"执行层是对的、只是组织得好不好"。本维专审**执行层做的事对不对**。
> 对带脚本 / 命令 / API 调用的工具型 skill,这比维度 3 还高频致命。纯指导型 skill 本维可标 N/A。

**① 检查问题**
- skill 调用的命令 / 脚本 / API,其语义**真的能达成它宣称的效果吗**?(尤其招牌功能的核心命令——用 `--help` / 实测核对一遍)
- **跨平台吗?** macOS BSD vs GNU 差异:`grep -P`、`sed -i`、`date`、`readlink -f` 等;JSON 用 `jq` 还是 `grep -oP`(后者 mac 无 `-P` 会静默失败)。
- 错误码、边界、超时、重试有没有处理?有没有 **silent-fail**(命令失败却不报错、继续往下跑)?

**② 好的样子**
- 核心命令经 `--help` / 实测验证;用可移植工具(`jq` 解析 JSON);失败 loud(非 0 退出 + 明确中文 error);幂等;stdout 用 `SUCCESS:`/`ERROR:` 前缀便于上游解析。

**③ 坏味道**
- 招牌功能的命令**语义用错**(例:`git pull origin x -- <subdir>` 假装能"只拉子目录",但 git pull 无 pathspec 语法,`--` 后被当 refspec;部分检出须靠 `sparse-checkout`)。
- 依赖 GNU-only flag 在 mac 上静默失败(`grep -oP` 读不到值还往下跑)。
- 吞错误继续;无超时;边界(空结果 / 0 命中)未处理。

**④ 严重度**:**招牌功能命令语义错 = 🔴**(架构再漂亮也没用,功能地基有洞);跨平台 silent-fail = 🟠;缺边界 / 超时 = 🟡。

## 专项:含反馈环 / 重试 / 回退的 skill

带"回退环 / 重试 / 自纠循环"的 skill 是独立形态,有专属失败模式。命中就逐条查:
- **终止性可证**:最坏情况几跳后停?能写出"最坏 N 跳 → 接受 / 失败"吗?
- **预算落盘**:回退计数器 / 已走边 / log 落盘了吗?**只存上下文 = 等于没存**(压缩后归零)。
- **计数口径唯一**:一次诊断结论消耗几次预算,说死了吗?(模糊 → 两个实例算出不同余额)
- **强制动作计入预算**:"强制再审 / 必重试"这类动作有没有占预算?(不占 → 绕过上限)
- **横跳 / 震荡封死**:每条边 ≤1 次 + 单调收紧(目标只能越来越靠上游),防 A→B→A→B。
- **耗尽出口**:耗尽是"接受最好"还是"failed"?(接受需落盘每次产出+评分才能选最好)

## 专项:方法论 / 标准型 skill(自举一致性)

当被审对象**自己就是一套标准 / 方法论 / 测试体系**(它对外立规则要求别人遵守),最致命的缺陷是**双标**——它要求别人做的,自己没做到。逐条查:
- **单一源头(前置,治根)**:总则 / 契约 / 评分锚点有没有声明**唯一 source-of-truth**?其余文档是 cross-ref 还是各存一份全文?**存全文 = 预判必漂移**(比"分叉后才发现"早一档)。实例:同一套总则散在 4 个文件 → 必然漂移。
- **自举一致**:它对外立的每条规则(记偏离 / 落盘 / 裁判分层 / 先有标准…),**自己的每个子文档都遵守了吗**?
- **无双标**:同一规则,文档 A 守、文档 B 不守?(实例:state-tests 有偏离格式、quality-rubric 没有)
- **总则不自漏**:总则列举某分层 / 清单时,有没有漏掉自己体系里的一项?(实例:"各层都有裁判"却漏列某层裁判)

> 这类病和"单体 video-remix 要求落盘、自己却拒绝落盘"**同构**:**对外严、对己松半档**。一套教别人的标准,自己不达标 → 可信度崩。
> **④ 严重度**:自举双标 🟠。

---

## 专项:重构 / 换名词式改造的遗留扫描

当被审对象是**改了核心名词 / 支柱的重构版**(如 `ref_profile`→`blueprint`、某状态字段改名、一条主线语义翻转),旧措辞最容易残留在边角并与新支柱冲突。四查高收益、可机检:
- **旧名词 grep 清零**:`grep -rn '<旧名词>'` 全 skill 目录,主文件应 0 残留(残留处往往是契约表 / example / 落盘文件名这种边角)。
- **跨文件 schema 字段一致**:新引入的字段名挨个 `grep -rl`,确认各文件**拼写一致 + 都出现**(防一处叫 `slot_fit`、另一处叫 `slot_match` 的 drift);枚举值(如 arc 角色集 HOOK/DEMO/…)也跨文件对齐。
- **醒目位逐条翻面**:北极星句(你是谁 / 质量标准)、原则、红旗 / 反模式表——这些是 LLM 最先读、压缩时最后丢的位置,旧理念残留在此危害最大,**必逐条翻面**(实例:质量标准还写"跟 ref 不违和",而新支柱是"内容可与 ref 不同域")。
- **新引用锚点防悬空**:重构新增的 cross-ref(`见 §X` / `按 Y 清单`)挨个核对**目标锚点真存在**;尤其"早阶段引用晚阶段才定义的清单" → 要么前置该清单、要么改引用(实例:Step 3 要标 fc_feasible 却引用埋在 Step 7 的表)。

> 与归类「重构后复审型」配套:归类决定**先修复回归 + 遗留扫描**,本专项给遗留扫描的**可机检四查**。
> **④ 严重度**:旧理念残留在醒目位 🟠;schema 字段 drift 🟠(下游静默读错字段);边角残留 / 悬空引用 🟡。

## 打分与输出

- **先归类**(长链路有状态 / 短链路工具 / 纯指导)→ 决定维度 3 与维度 8 谁是主战场。
- 逐维记录,缺陷给 `file:line` 证据 + 对照标尺。固定记录维度 2 的"不做什么边界"打勾位。
