# 缺环分析: in-chaos 生态互驱效果复盘

> 2026-06-01，基于 31 个会话、12 commits、~$0.93 消耗的第三方复盘。

## 一、现有效果 (做到位的)

| 做到什么 | 证据 |
|----------|------|
| 盲区互捕确定性高 | evaluating 发现 reviewing 自举双标, reviewing 发现 iteration "全部状态可入口"虚假声明, evaluating 的双副本漂移是 reviewing 审出来的 |
| 收敛趋势明确 | 缺陷严重度逐轮下降: 架构级 🔴 → 结构级 🟠 → 健壮性 🟡 → 抛光层 ⚪ |
| dogfooding 不自欺 | iteration 用自己的 state-machine-check 迭代自身,v1→v4 有真实结构改善 |
| 自举一致性落地 | evaluating 13 项 checklist 11/13 → 13/13,靠的不是理论而是实测修正 |

## 二、核心缺环

### 缺环 1: 零外部验证

**证据:** 31 个会话,所有 subagent 任务 = 审/测/迭代彼此。没有任何一个外部 skill 走过这个管线。  
**后果:** 不知道这套流程对非生态内 skill 是否有效。整个方法论可能是面向自身定制的过拟合。

### 缺环 2: 人类在环外旁观

**证据:** 所有审计和评测由 subagent 独立完成,人类只在会话级别看到结果。中间判断(缺陷是否为真、是否值得修)无人类确认。  
**后果:** agent 可能修了自己造的假问题(见 evaluating v0.6.1 修正行号偏移这类 polish)。

### 缺环 3: 零跨 session 测试

**证据:** 全部迭代在一天内跑完,无跨 session 续跑、上下文压缩后恢复的测试。  
**后果:** 正在审别人的维度 3 (状态跨上下文传递),自己从未被跨 session 测过。

### 缺环 4: 仪式过度,非均值压缩

**证据:** iteration: 8 状态 + log/preferences/techs/archive 五层目录; reviewing: 8 维 rubric + 4 类归 + 3 专项; evaluating: 3 层 + 6 总则 + 13 checklist。  
**后果:** ~$0.93 中有可观部分花在 CHANGELOG 行号校对、Rigid/Flexible 标签、cross-ref 回链这种非核心工作上。

### 缺环 5: 无"使用先于迭代"约束

**证据:** 三个 skill 在 v0.1 阶段就开始互驱迭代,没有一个经过外部真实负载使用。reviewing 的 CHANGELOG 表明 v0.1 仅从审视 video-remix 提炼,此后全是在自身生态内迭代。  
**后果:** 迭代速度 > 反馈速度。迭代引擎在空转。

## 三、缺环 → 补全

| 缺环 | 补全方案 | 实现位置 |
|------|----------|----------|
| 零外部验证 | 每轮互驱必须有至少一个外部被测 skill。审计完成后,人类确认报告是否对外部 skill 也成立 | SKILL.md §外部验证门 |
| 人类在环外旁观 | 审阅报告产出后→人类确认入口(只确认 🔴 和 🟠,⚪ 自动丢弃) | SKILL.md §人类确认门 |
| 零跨 session 测试 | 每轮必须跑一次跨 session 模拟: 关闭上下文→新 session 续跑→验证中间态 | SKILL.md §跨 session 门 |
| 仪式过度 | 只保留 4 维 rubric、2 层评测(L3 省略)、0 个 log/preferences/techs 子目录 | SKILL.md 整体架构 |
| 无"使用先于迭代"约束 | 无外部使用证据 → 不进迭代管线 | SKILL.md §入口条件 |

## 四、本质总结

三个 skill 的设计意图正确——盲区互捕 + 自举一致 + 生态内互驱。但**闭环太紧**:它们互为对方唯一的用户和被测对象。这既是收敛快的原因,也是外部有效性的根本隐患。

**一个判断**:这套方法论对 skill 作者有用,但需要"破圈"——至少一个外部 skill 的真实测试结果,才能验证方法论不是面向自身过拟合的。
