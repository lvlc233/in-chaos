# 懒人版 3-skill 互驱流程

> 从 in-chaos 生态 8 状态 + 8 维 + 3 层 + 13 checklist → 精简为 5 步 + 3 门。
> **目标**: 去仪式、留干货、补盲区。能用一个文件解决的不用三个目录。

## 角色定义

任意时刻,三个 skill 各占一个角色:

| 角色 | 做什么 | 谁可以做 |
|------|--------|----------|
| **审计者 (A)** | 读被测 skill 全文,用 4 维 rubic 出审阅报告 | reviewing-in-chaos |
| **评测者 (E)** | 测被测 skill 的动态行为,跑 + 记偏离 | evaluating-in-chaos |
| **迭代引擎 (I)** | 接收审阅报告 + 评测偏离,执行修复 | iteration-in-chaos |

## 4 维精简 rubric (A 用)

审查 skill 时只看这 4 个致命维度,不逐帧扫 8 维:

| 维度 | 核心问题 | 何时标 🔴 |
|------|----------|-----------|
| **D1 描述正确性** | description 写的是触发条件吗?和正文一致吗? | description 写的是能力清单而非触发条件 (LLM 会跳过正文) |
| **D2 跨上下文健壮性** | 中间态落盘吗?压缩后能续跑吗? | 长链路 + 不落盘 = 生产必然丢失状态 |
| **D3 失败模式** | 核心命令/API 语义对吗?失败了有兜底吗? | 招牌功能的命令语义错 (架构再漂亮也没用) |
| **D4 职责边界** | 管什么?明确不管什么?和别人冲突吗? | 多职责串在一张流程图内,无边界声明 |

> 如果被测 skill 是纯指导型(无脚本/命令), D4 替代 D3 为主战场。

## 2 层精简评测 (E 用)

| 层 | 测什么 | 最小方法 |
|----|--------|----------|
| **L1 行为正确性** | 读 skill 后的行动计划是否达预期 | 跑 3 次,取 pairwise diff,记偏离 |
| **L2 状态流中间质量** | 关键状态的中间产物是否和状态契约一致 | mock 单状态,注入好/坏上游产物,查产出 |

> 省略 L3 (内容/审美),懒人版不测"好不好看"。

## 5 步互驱流程

### Step 1: 选定被测 + 角色分配

```
[审查会话] 选本轮被测 skill T,分配 A 和 E
- T = reviewing-in-chaos  → A=evaluating, E=iteration
- T = evaluating-in-chaos  → A=iteration, E=reviewing
- T = iteration-in-chaos   → A=reviewing, E=evaluating
```

### Step 2: A 审 T → 出报告

A 执行:
1. 读 T 的 SKILL.md + 所有 references (逐行)
2. 对照 4 维 rubric,给每个缺陷 `file:line` + severity
3. 只报告 🔴 和 🟠; 🟡 和 ⚪ 记录但不出报告
4. 输出一句话根因

### Step 3: [人类确认门] 🚪

**此刻停,把 A 的 🔴/🟠 缺陷列表呈现给人类。人类确认:**
- 哪些是真缺陷?哪些是 agent 误判?
- 哪些缺陷本轮修?哪些 backlog?

人类未确认 → 不进 Step 4。

### Step 4: E 测 T → 记基线 + 修复回归

E 执行:
1. 测 T 当前行为 (L1 跑 3 次 + L2 mock 单状态)
2. 记录基线偏离

I 执行:
1. 按人类确认的缺陷列表逐项修复
2. 每修一个 → 在 T 的 CHANGELOG/TODO 追加一行 (日期/修复项/来源)

E 再跑:
1. 同上测试,对比基线 → 确认改善或检出回归

### Step 5: [外部验证门 + 跨 session 门] 🚪🚪

**外部验证:** 用 T 测一个非生态内 skill (不在 in-chaos 内的),记录是否发现真实问题。无外部 skill 可用 → 标 `⚠️外部验证未执行`,不阻塞但记录。

**跨 session 模拟:** 关闭当前上下文,新 session 内只给 agent 上一轮的中间产物,要求续跑。续跑成功 → ✅。失败 → 标注 🔴。

## 轮换规则

完成 Step 5 后,角色轮换:

```
上一轮: A 审 T, I 修 T
下一轮: 上一轮的 A 成为新的 T
即: 每次迭代的"被测对象"是上一轮的"审计者"
```

| 轮次 | T (被测) | A (审计) | E (评测) |
|------|----------|----------|----------|
| 1 | reviewing | evaluating | iteration |
| 2 | evaluating | iteration | reviewing |
| 3 | iteration | reviewing | evaluating |
| 4 | reviewing (回到 1) | ... | ... |

> 轮次和状态追踪写在 `cycle-log.md` 里,一行一个轮次。
