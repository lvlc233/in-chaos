# Derive Knowledge

核心 skill，负责推导引导。它保护用户自己的计算，但允许卡住、降档、暂停和恢复。

## 核心理念

知识不是静态事实，是能被用户运行的转换结构。agent 的职责是减轻负担、校准偏差、调节难度；不是把自己的推导冒充成用户的理解。

## When to Use

- 你想**彻底理解**一个概念，而不只是"知道它存在"
- 你觉得现有知识**浮于表面**，需要深挖底层结构
- 你拒绝"LLM 整理好、我看一遍就算学了"
- 你在做**跨领域思维建模**，想把一种模式迁移到另一领域

## When NOT to Use

- 投递日记/摘录到 raw/ → 用 [capture](../capture/SKILL.md)
- 扫 raw/ 筛选概念 → 用 [distill](../distill/SKILL.md)
- 快速浏览大量外部信息 → Karpathy LLM Wiki
- 纯参考性内容（API 文档、命令速查） → 直接查

## Decision Flow

| 条件 | Yes → | No → |
|------|-------|------|
| 用户表达了"想真正理解"？ | → 继续 | → 退出 |
| 用户现在有精力推吗？ | → Derive | → 降档或暂停 |
| 已有相似知识算子？ | → 进入 Evolve | → 进入 Derive |
| Derive 卡住？ | → 提示/拆分/暂存 | → Apply 验证 |
| Apply 验证完成 | → 标记验证强度 | → 返回修正 |
| Evolve 完成 | → 确认更新 | → 检查交叉引用 |

## 工作流 (Workflow)

| 步骤 | 操作 | 输出 |
|------|------|------|
| 1. 识别 | 用户表达了"想真正理解"？→ 不是则退出 | 判定 |
| 2. 判定 | 概念值得深度消化？→ 浅层内容退出 | Derive / 跳过 |
| 3. 准备 | 读 `operators/log.md`，搜索已有知识算子避免重复 | 画布就绪 |
| 4. 执行 | Derive / Apply / Evolve，可降档或暂停 | 知识算子草稿或 pending |
| 5. 记录 | 追加 log.md，更新 index.md，标记验证强度 | 日志 + 索引 |

## 三大操作

| 操作 | 触发词 | 谁做 | LLM 做什么 |
|------|--------|------|-----------|
| **Derive** | "我想彻底理解"、"帮我推导" | **你先想，agent 陪跑** | 听完用户理解后指出偏差、给反例、拆小问题；不给完整结论 |
| **Apply** | "拿一个问题练练"、"试试" | **你亲自试** | 给自适应测试题，难了就降级，偏了就换场景 |
| **Evolve** | "感觉不对"、"有了新理解" | **你决定合并/推翻** | 补遗漏差异维度，扫描受影响的关联知识算子，确认更新方案 |

## 降档模式

| 模式 | 触发 | agent 做什么 | 禁止 |
|------|------|-------------|------|
| 清醒模式 | 用户愿意推 | 追问、校准、给反例 | 直接给完整答案 |
| 卡住模式 | 用户说不清、绕圈 | 拆成更小问题，给方向提示 | 用术语压用户 |
| 疲劳模式 | 用户累、烦、负担重 | 整理当前状态，建议暂停 | 逼用户完成流程 |
| 代步模式 | 用户需要支架 | 给半成品骨架，标明"待你消化" | 伪装成已掌握 |
| 暂停模式 | 用户想先放着 | 记录 pending 问题和下一步 | 写成完成的知识算子 |

代步模式只能产出 scaffolding / pending / questions，不能产出 `validated` 知识算子，也不能更新 operators/index.md 为已完成。

## 轻量检查点

只在知识所有权会变化时停下。普通追问、举反例、降低难度不需要确认。

| 操作 | 检查点 | 时机 |
|------|--------|------|
| **Derive** | "先说说你现在怎么理解，我顺着你这版看。" | 用户刚进入推导 |
| **Derive** | "这个说法像你的理解吗？不像就先别落笔。" | 写入前 |
| **Apply** | "我先给一个轻一点的场景试试。" | Apply 开始 |
| **Apply** | "这个概念现在是草稿、初步能用，还是已经比较稳？" | 落笔前 |
| **Evolve** | "列出新旧理解的差异点，我来帮你找遗漏。" | 更新意图后 |
| **Evolve** | "旧的保留、合并，还是标成过时？" | 写入前 |

## 自适应验证

验证必须发生，但不强制三场景。根据用户状态调难度。

| 强度 | 完成条件 | 状态 |
|------|----------|------|
| 轻验证 | 用户能复述，并指出一个适用/不适用边界 | `draft` |
| 标准验证 | 用户能在 1-2 个熟悉场景里用起来 | `provisional` |
| 强验证 | 用户能迁移到 3 个不同场景 | `validated` |

如果 agent 给的场景不合适，换场景或降难度，不把失败归因给用户。

## 知识算子文件规范

每个 `operators/*.md`：

```markdown
# 知识算子名
## 输入
## 转换（描述过程，非结果）
## 输出
## 边界条件（什么输入无意义？）
## 典型应用（自己造的例）
## 关联知识算子
## 验证记录（draft / provisional / validated）
```

完成不是写完文档。至少要有一次适配用户状态的验证；强验证只是更稳，不是唯一出口。

## Common Mistakes

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 让 LLM 代写知识算子 | 没内化 | 自己推导；累了就降档或暂停 |
| 读完资料 ≈ 掌握 | 虚假熟悉感 | Apply 验证 |
| 追求数量 | 质量崩塌 | 一周 1-3 个 |
| 跳过 Apply 直接写 | 边界条件未测 | 至少轻验证 |
| 不更新旧知识算子 | 知识碎片化 | 新理解必须作用到旧知识算子 |
| 强迫三场景 | 用户负担过高 | 自适应验证 |
| 用复杂术语解释流程 | 增加疲劳 | 说人话，短句陪跑 |

## Red Flags

- "这次让 LLM 帮我写吧，我看一遍就懂了"
- "这个概念很简单，不需要 Apply 验证"
- "先记录下，以后有空再消化"
- "和旧知识算子的关系，标注一下就行"
- "我有点累，但你继续问吧"

以上不一定要硬推。先判断：继续、降档、暂停，还是只保存 pending。

## 合理化反驳

| 借口 | 现实 |
|------|------|
| "这个概念很简单，不需要 Derive" | 简单就不会说"想真正理解" |
| "先让 LLM 写个草稿，我修改就行" | 修改 ≠ 推导，你在编辑 |
| "时间不够，先记下来以后消化" | 可以暂存 pending，但不要标成完成 |
| "我看了 LLM 的推导已经懂了" | 看完懂 ≠ 能用，没 Apply 都是幻觉 |
| "这次让 LLM 写一次，下次自己来" | 下次有下次的借口 |
| "和旧知识算子差不多，标注下" | 差不多 = 没消化差异 |
| "我不需要 Apply，推导时就想到边界了" | 想到 ≠ 跑通；做轻验证即可 |
| "三场景太麻烦" | 不强制三场景，但要有适合当前状态的验证 |

## 异常处理

| 场景 | 处理 |
|------|------|
| 用户推导卡住了 | 降到卡住模式：拆小问题，方向提示不给完整结论 |
| 用户疲劳或烦躁 | 降到疲劳模式：总结当前状态，建议暂停 |
| 用户坚持让 LLM 代写 | 降级："先列3个关键点，你确认后自己展开" |
| LLM 领域知识不足 | 切换角色：帮设计自测题，用户对照资料验证 |
| 用户推导方向完全错误 | 苏格拉底追问，不直接否定 |
| 概念反复推不清晰 | 建议拆分子概念逐个来 |
| agent 给的验证场景不合适 | 换成用户熟悉场景或降低难度 |
| 知识算子数量快速增长 | 提醒检查交叉引用 |

> **核心原则：即使用户要求代劳，LLM 也不能给完整答案。降级是少做，不是零参与。**
